# 推理小说分步创作系统设计

这个系统不是一个全自动写完整本小说的工具，而是一个由用户确认驱动的分步创作工具。

用户先输入关键词，LLM 只生成当前阶段需要用户判断的内容。用户确认后，系统才把本阶段结果锁定到案件档案，并进入下一阶段。

核心原则：

```text
每一步只生成下一步所需内容；没有用户确认，不进入下一阶段。
```

## 1. 产品范围

第一版只做单本推理小说的最小闭环：

- 用户输入关键词和基础偏好。
- LLM 生成案件真相候选。
- 用户确认案件真相。
- LLM 基于已确认真相生成时间线、人物、线索。
- 用户逐步确认这些结构化设定。
- LLM 基于已确认设定生成章节大纲。
- 用户确认章节大纲。
- LLM 按章节生成正文。
- 系统做最小逻辑检查，发现不确定或冲突时回到用户确认。

暂不做：

- 完全自动生成整本书。
- 多角色自动调度框架。
- 复杂看板。
- 多案件单元剧。
- 超长连载管理。
- 向量库、Redis、任务队列。

第一版先证明一件事：系统能不能让用户一步步确认并沉淀一个严谨、可复盘的推理案件。

## 2. 用户流程

```text
关键词输入
  -> 案件真相生成
  -> 用户确认案件真相
  -> 时间线、人物、线索生成
  -> 用户确认结构化设定
  -> 章节大纲生成
  -> 用户确认章节大纲
  -> 逐章正文生成
  -> 逻辑检查与修订
```

所有阶段都遵守同一个规则：

```text
生成草案 -> 用户修改或确认 -> 写入版本 -> 进入下一步
```

LLM 不能因为“看起来合理”就自动推进。涉及真相、动机、时间线、关键线索、章节功能的变化，都必须让用户确认。

## 3. 阶段设计

### 3.1 关键词输入

用户在第一页输入创作关键词和约束，例如：

```text
现代、刑警、连环杀人、宗教仪式、曲折、反转、不写超自然真相
```

这一页只收集用户意图，不生成完整设定。

最小字段：

```json
{
  "keywords": ["现代", "刑警", "连环杀人", "宗教仪式", "反转"],
  "style": "冷峻、社会派、本格结合",
  "length": "30章左右",
  "limits": ["不写超自然真相"]
}
```

如果关键词互相冲突或不够明确，系统直接提问，不要硬生成。

### 3.2 案件真相

LLM 根据关键词生成 1 到 3 个案件真相方案。这里的“真相”必须直接展示给用户确认，不要把真凶和核心反转藏在系统内部。

每个方案至少包含：

- 被害人是谁。
- 表层谜面是什么。
- 真凶是谁。
- 真凶动机是什么。
- 真实作案过程是什么。
- 凶手如何掩盖真相。
- 最终反转是什么。
- 这个方案的推理卖点是什么。

用户可以：

- 选择一个方案。
- 修改局部设定。
- 要求重新生成。
- 回答系统提出的澄清问题。

确认后，案件真相进入锁定状态。后续阶段不能静默改动真凶、动机、作案方式和最终反转。

### 3.3 时间线、人物、线索

案件真相确认后，LLM 才生成支撑真相的结构化设定。

这一阶段生成三类内容：

| 内容 | 目的 |
| --- | --- |
| 时间线 | 确认真实发生顺序、角色声称顺序、发现顺序 |
| 人物 | 确认嫌疑人、证人、被害人、主角及其关系 |
| 线索 | 确认哪些线索公平出现、如何误导、何时回收 |

这一阶段可以拆成多个确认页，不要一次把所有设定塞给用户。

建议顺序：

```text
真实时间线
  -> 人物与动机
  -> 关键线索
  -> 误导与红鲱鱼
```

每一页只确认一个层级。系统发现缺口时直接问用户，例如：

```text
凶手在 21:00 到 21:20 之间缺少可执行作案窗口。要把案发时间提前，还是给凶手增加伪造不在场证明？
```

### 3.4 章节大纲

时间线、人物和线索确认后，LLM 生成章节大纲。

章节大纲不是自由剧情，而是信息释放计划。每章必须说明：

- 本章目的。
- 本章出现的新线索。
- 本章制造的误导。
- 本章不能泄露的信息。
- 本章推进的人物关系。
- 本章是否回收前文线索。

默认先支持 30 到 60 章，不做无限长篇。

### 3.5 正文生成

正文生成只能基于已确认内容。

硬规则：

1. 不新增未经确认的关键证据。
2. 不修改已确认案件真相。
3. 不让侦探知道尚未发现的信息。
4. 不泄露本章 `must_not_reveal`。
5. 不用巧合破案。
6. 不让角色无理由说谎。
7. 新增会影响后续推理的事实，必须回到用户确认。

正文可以逐章生成。用户修改章节正文时，只有影响案件逻辑的修改才需要更新案件档案。

## 4. 状态模型

第一版只维护一个 `NovelState`。不要拆复杂表，也不要引入工作流引擎。

```json
{
  "stage": "case_truth",
  "brief": {
    "keywords": ["现代", "刑警", "连环杀人", "宗教仪式", "反转"],
    "style": "冷峻、社会派、本格结合",
    "length": "30章左右",
    "limits": ["不写超自然真相"]
  },
  "case_truth": {
    "victim": "被害人",
    "surface_mystery": "读者最初看到的谜面",
    "culprit": "真凶",
    "motive": "真实动机",
    "method": "真实作案方式",
    "cover_up": "掩盖方式",
    "final_twist": "最终反转",
    "status": "confirmed"
  },
  "timeline": [],
  "characters": [],
  "clues": [],
  "chapters": [],
  "confirmations": [],
  "quality_reports": []
}
```

`stage` 决定当前页面和下一步可执行动作。第一版只需要这些阶段：

```text
brief_input
case_truth
case_truth_confirmed
case_structure
case_structure_confirmed
chapter_plan
chapter_plan_confirmed
drafting
revision
```

## 5. 最小字段

### 5.1 确认记录

```json
{
  "stage": "case_truth",
  "status": "confirmed",
  "summary": "真凶是被害人的旧案搭档，宗教仪式是伪装，真实动机是掩盖十五年前的伪证。",
  "locked_fields": [
    "case_truth.culprit",
    "case_truth.motive",
    "case_truth.method",
    "case_truth.final_twist"
  ],
  "created_at": "2026-07-08T12:00:00Z"
}
```

`locked_fields` 后续不能静默改。需要修改时，系统必须提示影响范围，并生成新版本。

### 5.2 人物

```json
{
  "name": "林澈",
  "role": "刑警 / 嫌疑人 / 被害人 / 证人",
  "relationship": "与核心案件的关系",
  "motive": "表面动机或真实动机",
  "secret": "隐藏信息",
  "lie": "说谎点",
  "truth_status": "culprit / innocent / accomplice / victim / witness"
}
```

### 5.3 时间线

```json
{
  "time": "21:35",
  "location": "地下停车场",
  "actual_event": "真实发生的事",
  "claimed_event": "角色声称发生的事",
  "people": ["真凶", "被害人"],
  "reader_knows_at": "第几章",
  "detective_knows_at": "第几章"
}
```

### 5.4 线索

```json
{
  "id": "c001",
  "description": "未发送短信",
  "first_seen": "第3章",
  "surface_meaning": "看似指向前妻",
  "real_meaning": "实际指向备用手机",
  "payoff": "第26章",
  "fair": true
}
```

### 5.5 章节

```json
{
  "chapter": 12,
  "purpose": "强化错误嫌疑人，同时埋下真相线索",
  "new_clues": ["c001"],
  "misdirection": "让读者怀疑前妻",
  "must_not_reveal": "备用手机属于真凶"
}
```

## 6. 页面设计

第一版不要做复杂工作台。按阶段给用户一个清晰的确认流。

### 6.1 页面列表

| 页面 | 用户动作 | 系统动作 |
| --- | --- | --- |
| 关键词页 | 输入关键词和限制 | 创建 `brief` |
| 案件真相页 | 选择、修改或重生成真相 | 锁定 `case_truth` |
| 结构设定页 | 确认时间线、人物、线索 | 写入结构化案件档案 |
| 章节大纲页 | 调整章节功能和信息释放 | 锁定章节计划 |
| 正文页 | 逐章生成、编辑、确认 | 保存章节正文 |
| 复盘页 | 查看问题并选择修订方案 | 生成新版本 |

### 6.2 页面操作

每个确认页只放四类操作：

| 操作 | 含义 |
| --- | --- |
| 确认并继续 | 锁定本阶段结果，进入下一阶段 |
| 修改要求 | 用户补充偏好，基于当前结果修订 |
| 重新生成 | 丢弃当前候选，重新生成本阶段内容 |
| 提问/回答 | 系统或用户补足不确定信息 |

不要用一个“自动完成全部”的按钮替代阶段确认。

## 7. API

第一版 API 按阶段推进即可。不要为每个小字段设计独立接口。

```text
POST /api/novels
POST /api/novels/:id/generate
POST /api/novels/:id/confirm
POST /api/novels/:id/revise
GET  /api/novels/:id
```

### 7.1 生成

```json
{
  "stage": "case_truth",
  "feedback": "更偏社会派，但保留本格谜面"
}
```

服务端根据 `stage` 和当前 `NovelState` 只生成当前阶段内容。

### 7.2 确认

```json
{
  "stage": "case_truth",
  "decision": "确认方案 B，但把宗教仪式改成凶手伪装",
  "locked_fields": [
    "case_truth.culprit",
    "case_truth.motive",
    "case_truth.method",
    "case_truth.final_twist"
  ]
}
```

服务端做四件事：

1. 读取当前版本。
2. 校验用户确认的阶段是否等于当前 `stage`。
3. 写入 `confirmations`，更新 `locked_fields`。
4. 生成新版本并推进到下一阶段。

## 8. 存储

沿用当前轻量栈：

- Vue：分步确认界面。
- Hono/Cloudflare Worker：生成和确认 API。
- Cloudflare D1：保存小说项目、当前版本、章节索引。
- Cloudflare R2：保存 `NovelState` 快照、章节正文和导出文件。
- LLM Structured Output：约束每阶段输出格式。

最小存储模型：

```text
D1
  novels(id, title, current_version, stage, created_at, updated_at)
  versions(id, novel_id, stage, r2_key, created_at)
  chapters(id, novel_id, chapter_no, title, status, r2_key, updated_at)

R2
  novels/{novel_id}/state/{version}.json
  novels/{novel_id}/chapters/{chapter_no}/{version}.md
  novels/{novel_id}/exports/{version}.md
```

每次确认或生成正文都产生新版本，不覆盖旧版本。

## 9. 逻辑检查

每次确认案件真相、结构设定、章节大纲后，都跑一次最小检查。

```json
{
  "pass": false,
  "questions": [
    "凶手在 21:00 到 21:20 之间如何离开监控区域？"
  ],
  "problems": [
    {
      "severity": "high",
      "issue": "第29章推理依赖备用手机，但备用手机第一次出现也是第29章。",
      "fix": "在第7章现场勘查时加入备用手机的异常信号记录。"
    }
  ]
}
```

检查只做这些事：

- 时间线能否对上。
- 凶手行动路径是否可行。
- 关键线索是否提前出现。
- 每条关键线索是否有回收。
- 红鲱鱼是否解释清楚。
- 侦探推理是否跳步。
- 证据链是否过弱。
- 反转是否推翻前文事实。

如果检查结果里有无法自行判断的问题，系统必须问用户，不要自动补设定。

## 10. 结论

第一版的核心不是“让系统自动写完”，而是这个闭环：

```text
用户确认 -> 锁定当前阶段 -> 生成下一阶段 -> 检查问题 -> 必要时提问
```

这个闭环跑通后，再考虑更复杂的自动化、看板和长篇管理。
