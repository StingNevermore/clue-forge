# 现代推理小说 Agent 系统设计

本文档描述一个面向现代推理小说的 Agent 系统。它的目标不是优先生成更多正文，而是先保证案件逻辑成立、线索公平、误导合理，并让读者能够在最终揭示后复盘真相。

## 1. 设计目标

| 类型 | 核心关注 |
| --- | --- |
| 长篇网文 | 爽点、升级、连载节奏、人物成长 |
| 现代推理小说 | 谜面、公平线索、误导、逻辑闭环、时间线、证据链、真相反转 |

因此，本系统不应优先做“章节续写”，而应先建设：

- 案件设计系统
- 线索管理系统
- 逻辑一致性质检系统

推理小说 Agent 的主要目标是：

1. 谜题成立。
2. 线索公平。
3. 误导合理。
4. 犯案手法可执行。
5. 人物动机可信。
6. 时间线无漏洞。
7. 最终揭示能够回扣前文。

系统必须避免以下问题：

- 作者在结尾强行解释。
- 关键线索没有提前埋设。
- 凶手临时变更。
- 作案手法不现实。
- 侦探依赖直觉开挂。
- 读者无法复盘。
- 时间线无法对齐。

核心原则：先设计真相，再倒推故事。

```text
真相 -> 犯案过程 -> 证据残留 -> 误导路径 -> 侦查过程 -> 读者体验
```

## 2. 核心架构

系统由 7 个核心 Agent 组成：

```text
案件设计 Agent
  -> 人物关系 Agent
  -> 时间线 Agent
  -> 线索/证据 Agent
  -> 误导/红鲱鱼 Agent
  -> 章节写作 Agent
  -> 公平性与逻辑质检 Agent
```

与普通小说不同，推理小说不适合从第一章开始顺写。系统应先确定真实案件，再倒推出谜面、误导、调查路径和章节信息节奏。

## 3. 核心数据结构

系统需要维护一个“案件 Bible”。它类似普通小说的作品 Bible，但更强调逻辑、证据和信息可见性。

### 3.1 Case Bible

```json
{
  "case_id": "case_001",
  "setting": "现代都市",
  "case_type": "谋杀 / 失踪 / 绑架 / 诈骗 / 旧案重查",
  "victim": "被害人",
  "culprit": "真凶",
  "surface_mystery": "读者最初看到的谜面",
  "real_truth": "案件真实经过",
  "core_trick": "核心诡计",
  "theme": "案件背后的主题",
  "detective_role": "刑警 / 私家侦探 / 记者 / 普通人 / 法医",
  "narrative_pov": "第一人称 / 第三人称 / 多视角"
}
```

其中最重要的是：

- `surface_mystery`：表层谜面。
- `real_truth`：真实真相。
- `core_trick`：核心诡计。

推理小说必须明确区分“读者看到的谜面”和“案件真实发生的过程”。

### 3.2 Suspect

每个嫌疑人都应有结构化信息。

```json
{
  "name": "林某",
  "relationship_to_victim": "前合伙人",
  "motive": "经济纠纷",
  "opportunity": "案发时在现场附近",
  "alibi": "声称在酒吧，有监控为证",
  "secret": "其实替别人隐瞒行踪",
  "red_herring_role": "强嫌疑但非真凶",
  "truth_status": "innocent / culprit / accomplice / witness",
  "what_they_lie_about": "不在场证明中的时间",
  "why_they_lie": "保护情人"
}
```

每个主要嫌疑人最好具备：

- 一个表面动机。
- 一个隐藏秘密。
- 一个说谎点。
- 一个被误解的行为。

这些要素不一定指向真凶，但必须能够在结尾解释清楚。

### 3.3 Timeline Event

时间线是推理小说 Agent 的核心数据。

```json
{
  "event_id": "e023",
  "actual_time": "21:35",
  "public_claimed_time": "21:50",
  "location": "地下停车场",
  "characters_present": ["真凶", "被害人"],
  "actual_event": "真凶与被害人发生争执",
  "known_to_reader_chapter": 8,
  "known_to_detective_chapter": 10,
  "source_of_information": "停车场监控时间戳",
  "truth_level": "true / false / partial / misleading"
}
```

时间线必须区分：

- 真实发生时间。
- 角色声称时间。
- 读者知道时间。
- 侦探知道时间。

这是推理小说和普通小说最大的差异之一。

### 3.4 Clue

线索必须管理完整生命周期。

```json
{
  "clue_id": "c014",
  "description": "被害人手机上最后一条未发送短信",
  "first_appearance_chapter": 3,
  "surface_meaning": "像是指向前妻",
  "real_meaning": "其实是被害人在提醒密码",
  "clue_type": "physical / testimony / digital / behavioral / contradiction",
  "fairness_level": "fair / subtle / hidden / unfair",
  "reader_can_infer": true,
  "detective_interpretation": "误以为是死亡前留言",
  "final_payoff_chapter": 18,
  "status": "planted / noticed / misread / reinterpreted / resolved"
}
```

每条关键线索都要明确：

- 表层含义。
- 真实含义。
- 首次出现章节。
- 最终回收章节。
- 读者是否有机会推理出来。

这决定线索是否公平。

### 3.5 Contradiction

推理小说不应依赖灵感破案，而应通过矛盾推进调查。

```json
{
  "contradiction_id": "ct007",
  "statement_a": "嫌疑人称 21:30 已离开公司",
  "statement_b": "电梯刷卡记录显示 21:42 仍在公司",
  "detected_by": "侦探",
  "chapter_detected": 9,
  "possible_explanations": [
    "他说谎",
    "有人使用了他的卡",
    "系统时间被修改"
  ],
  "true_explanation": "有人使用了他的卡制造误导"
}
```

系统应主动生成和追踪矛盾点，而不是只生成剧情。

## 4. 倒推式工作流

推荐工作流如下：

1. 确定案件真相。
2. 设计核心诡计。
3. 设计真凶动机。
4. 设计真实时间线。
5. 设计表层谜面。
6. 设计嫌疑人和假线索。
7. 设计证据链。
8. 设计侦探调查路径。
9. 设计章节结构。
10. 写正文。
11. 检查公平性和逻辑漏洞。

核心原则：先有真相，再有谜面。不要让 Agent 从“第一章发现尸体”开始自由发挥。

## 5. Agent 职责

### 5.1 Case Architect Agent

负责案件底层设计，只设计案件，不写正文。

```json
{
  "real_truth": "案件真实经过",
  "culprit": "真凶",
  "motive": "真实动机",
  "method": "作案方式",
  "cover_up": "掩盖方式",
  "key_vulnerability": "真相中的破绽"
}
```

### 5.2 Timeline Builder Agent

负责生成真实时间线和表层时间线。

```json
{
  "actual_timeline": [],
  "claimed_timeline": [],
  "reader_discovery_order": [],
  "detective_discovery_order": []
}
```

它要检查：

- 谁在什么时间在哪里。
- 谁看到了什么。
- 谁说了谎。
- 谁误解了什么。

### 5.3 Clue Designer Agent

负责线索设计。每个关键真相至少需要 2 到 3 条线索支撑。

示例：

```text
真凶进入过现场
  -> 线索 1：门禁记录异常
  -> 线索 2：鞋底有特殊清洁剂
  -> 线索 3：证词里提到了一句不该知道的话
```

多线索支撑可以避免最终揭示显得突兀。

### 5.4 Red Herring Agent

负责设计误导。好的红鲱鱼要满足：

- 表面合理。
- 逻辑可解释。
- 最终能自洽。
- 不欺骗读者视角。
- 不依赖作者故意隐瞒关键信息。

```json
{
  "red_herring": "前妻有明显杀人动机",
  "supporting_clues": [
    "她案发前和被害人大吵",
    "她账户收到一笔可疑转账"
  ],
  "real_explanation": [
    "争吵是因为孩子抚养权",
    "转账是被害人补偿款"
  ],
  "chapters_active": "3-11"
}
```

### 5.5 Detective Path Agent

负责设计侦探如何逐步接近真相，避免“侦探突然知道”。

典型调查路径：

```text
发现线索 -> 产生假设 -> 假设被证伪 -> 出现新矛盾 -> 重新解释旧线索 -> 锁定关键问题 -> 最终推理
```

状态流：

```text
unknown -> suspected -> investigated -> contradicted -> reinterpreted -> proven
```

### 5.6 Chapter Planner Agent

负责章节分配。现代推理小说的章节不应平均推进，而应控制信息释放节奏。

30 章中篇结构示例：

| 章节 | 功能 |
| --- | --- |
| 1-3 | 案发与谜面 |
| 4-8 | 初步调查，多个嫌疑人登场 |
| 9-12 | 第一个强嫌疑人 |
| 13-16 | 证据反转，第一个推论失败 |
| 17-21 | 旧关系和隐藏动机浮出 |
| 22-25 | 关键矛盾出现 |
| 26-28 | 重新解释所有线索 |
| 29 | 真相揭示 |
| 30 | 余波与主题落点 |

### 5.7 Logic Checker Agent

负责公平性和逻辑质检，是系统中最重要的 Agent。

它要检查：

- 真凶是否有作案时间。
- 作案方式是否可执行。
- 证据链是否足以支持结论。
- 读者是否提前获得关键线索。
- 侦探推理是否跳步。
- 嫌疑人的谎言是否有动机。
- 红鲱鱼是否最终解释清楚。
- 是否存在未回收线索。
- 是否存在时间线冲突。
- 是否有角色知道不该知道的信息。

```json
{
  "logic_score": 84,
  "fairness_score": 78,
  "problems": [
    {
      "severity": "high",
      "issue": "第 12 章侦探推断凶手使用备用钥匙，但备用钥匙第一次出现是在第 18 章，属于不公平线索。",
      "fix": "在第 5 章现场勘查时加入物业备用钥匙登记册。"
    }
  ],
  "can_publish": false
}
```

## 6. 质检维度

推荐评分体系如下：

| 维度 | 权重 |
| --- | ---: |
| 案件逻辑闭环 | 25 |
| 线索公平性 | 20 |
| 时间线一致性 | 20 |
| 人物动机可信度 | 15 |
| 误导质量 | 10 |
| 阅读节奏 | 5 |
| 文风完成度 | 5 |

推理小说中文风仍然重要，但在 Agent 系统里，逻辑优先级应高于正文润色。

## 7. 与长篇网文 Agent 的区别

| 模块 | 长篇网文 | 现代推理 |
| --- | --- | --- |
| 核心驱动 | 爽点、升级、冲突 | 谜题、线索、逻辑 |
| 规划方式 | 顺推为主 | 倒推为主 |
| 数据重点 | 人物成长、伏笔、战力 | 时间线、证据、证词、矛盾 |
| 质检重点 | 节奏、爽点、连续性 | 公平性、可复盘、逻辑闭环 |
| 章节目标 | 每章有爽点和钩子 | 每章释放、扭曲或重解释信息 |
| 主要风险 | 注水、战力崩坏、重复 | 强行解释、不公平、逻辑漏洞 |

## 8. 推荐数据库设计

最小 MVP 可以包含以下表：

- `novels`
- `cases`
- `characters`
- `suspects`
- `chapters`
- `timeline_events`
- `clues`
- `testimonies`
- `contradictions`
- `red_herrings`
- `investigation_steps`
- `quality_reports`

其中最重要的是：

- `timeline_events`
- `clues`
- `testimonies`
- `contradictions`

这四类数据决定推理小说是否成立。

### 8.1 Clue 状态机

```text
planned -> planted -> noticed -> misinterpreted -> reinterpreted -> resolved
```

### 8.2 Suspect 状态机

```text
introduced -> suspected -> cleared / escalated -> explained
```

### 8.3 Investigation 状态机

```text
hypothesis_created -> evidence_collected -> contradiction_found -> hypothesis_revised -> conclusion_supported
```

## 9. 章节生成流程

单章生成建议流程：

1. 读取案件 Bible。
2. 读取真实时间线。
3. 读取当前读者已知信息。
4. 读取当前侦探已知信息。
5. 读取本章要释放的线索。
6. 读取本章要强化的误导。
7. 生成章纲。
8. 检查是否泄露过多真相。
9. 写正文。
10. 检查逻辑与公平性。
11. 抽取新增线索、证词和矛盾。
12. 更新案件状态。

每章都要明确：

```json
{
  "chapter_no": 12,
  "chapter_function": "强化错误嫌疑人，同时埋下真相线索",
  "new_clues": ["c014", "c021"],
  "red_herrings_to_strengthen": ["rh003"],
  "hypothesis_to_break": "前妻作案假设",
  "reader_should_believe": "前妻嫌疑最大",
  "reader_should_not_yet_know": "真凶使用了受害人的备用手机"
}
```

其中 `reader_should_believe` 和 `reader_should_not_yet_know` 是关键信息控制字段。

## 10. Prompt 设计重点

### 10.1 正文写作 Agent

正文写作 Agent 的 Prompt 应包含以下硬性规则：

```text
你正在写现代推理小说。

硬性规则：
1. 不得泄露未到揭示阶段的真相。
2. 不得让侦探知道数据库中标记为“侦探未知”的信息。
3. 不得新增未经规划的关键证据。
4. 不得用巧合推动破案。
5. 不得让角色无理由说谎。
6. 不得制造无法回收的悬念。
7. 本章必须完成 chapter_function。
8. 所有新增线索必须写入 state_changes。
```

### 10.2 质检 Agent

质检 Agent 的 Prompt 应明确检查项：

```text
你是推理小说逻辑审稿人。

请检查：
1. 是否有时间线冲突。
2. 是否有角色知道不该知道的信息。
3. 是否有未铺垫的推理跳跃。
4. 是否有不公平线索。
5. 是否有误导读者但无法解释的描写。
6. 是否有违反现代现实逻辑的作案方式。
7. 是否有证据链不足的问题。
```

## 11. 推荐产品范围

第一版建议只做“单案件长篇”，不要一开始做多案件结构。

### 11.1 单案件长篇

一本书围绕一个核心案件展开。

适合：

- 社会派推理。
- 悬疑推理。
- 刑侦长篇。
- 旧案重查。
- 多年前秘密。

优点：

- 结构集中。
- 线索管理清晰。
- 适合 MVP。

### 11.2 单元案件 + 主线长篇

每 20 到 40 章一个案件，同时维护主线。

适合：

- 刑侦队。
- 侦探事务所。
- 法医主角。
- 记者主角。

优点是连载性强、商业化更好；缺点是系统复杂度明显更高。每个案件都需要独立的 Case Bible，同时还要维护全书主线。

推荐第一版范围：

```text
单案件 30 到 60 章现代推理小说 Agent
```

不要直接做 500 章刑侦单元剧。

## 12. MVP 落地范围

第一版只做以下能力：

1. 创建案件 Bible。
2. 生成嫌疑人表。
3. 生成真实时间线。
4. 生成线索表。
5. 生成章节大纲。
6. 生成正文。
7. 逻辑质检。
8. 自动修订。

MVP 输入示例：

```text
现代都市 / 女性失踪案 / 主角是刑警 / 风格偏社会派
```

MVP 输出：

- 案件真相。
- 嫌疑人关系。
- 真实时间线。
- 线索矩阵。
- 30 章章节规划。
- 第 1 章正文。
- 质检报告。

## 13. 高价值能力

现代推理 Agent 最有价值的功能不是“写正文”，而是以下检查能力。

### 13.1 时间线冲突检查

示例：

```text
A 说 21:00 在家。
但 A 在 20:55 出现在公司电梯。
公司到家最少需要 30 分钟。
因此，A 的证词存在冲突。
```

### 13.2 线索公平性检查

示例：

```text
最终推理依赖“备用手机”。
但备用手机第一次出现是在揭示章。
因此，该线索不公平。
```

### 13.3 证据链检查

示例：

```text
目前证据只能证明嫌疑人撒谎。
不能证明他是凶手。
因此，结论过强。
```

### 13.4 信息可见性检查

示例：

```text
第 8 章侦探提到“监控被剪辑过”。
但这个事实第 11 章才由技术员发现。
因此，侦探提前知道了未知信息。
```

### 13.5 红鲱鱼回收检查

示例：

```text
前妻账户转账作为强误导出现。
但最终没有解释来源。
因此，该误导未回收。
```

这些能力比单纯生成文本更有商业价值。

## 14. 产品形态

推荐产品形态是“案件看板”：

| 区域 | 内容 |
| --- | --- |
| 左侧 | 案件真相、时间线、嫌疑人 |
| 中间 | 章节大纲、正文编辑器 |
| 右侧 | 线索状态、误导状态、逻辑警告 |
| 底部 | 生成本章、检查公平性、检查时间线、重新埋线索 |

适合提供以下视图：

- 真实时间线视图。
- 读者已知信息视图。
- 侦探已知信息视图。
- 嫌疑人嫌疑值曲线。
- 线索埋设-回收矩阵。
- 章节信息释放曲线。

线索埋设-回收矩阵示例：

| 线索 | 首次出现 | 被误读 | 重新解释 | 最终回收 |
| --- | --- | --- | --- | --- |
| 未发送短信 | 第 3 章 | 第 5 章 | 第 16 章 | 第 28 章 |
| 门禁记录 | 第 4 章 | 第 9 章 | 第 19 章 | 第 27 章 |
| 鞋底清洁剂 | 第 7 章 | 第 12 章 | 第 23 章 | 第 29 章 |

## 15. 实现路线

### 15.1 第一阶段：案件生成器

先不要写正文，只做：

- 输入题材。
- 生成案件真相。
- 生成嫌疑人表。
- 生成时间线。
- 生成线索表。
- 生成误导表。

这个阶段用于验证推理逻辑是否成立。

### 15.2 第二阶段：逻辑检查器

实现以下检查：

- 时间线检查。
- 公平性检查。
- 证据链检查。
- 信息可见性检查。
- 未回收线索检查。

这是系统的核心差异化能力。

### 15.3 第三阶段：章节规划器

实现：

- 30 章章节结构。
- 每章释放哪些信息。
- 每章制造哪些误导。
- 每章让侦探形成或推翻什么假设。

### 15.4 第四阶段：正文生成器

最后再写正文。推理小说正文是呈现层，底层案件逻辑才是核心。

## 16. 极简技术方案

MVP 可以使用：

- PostgreSQL：存储结构化案件数据。
- pgvector：检索相似章节和风格片段。
- Redis/BullMQ：处理生成任务。
- Node.js/Bun：后端服务。
- React：案件看板。
- LLM Structured Output：约束所有 Agent 输出 JSON。

核心 API：

```text
createCaseBible()
generateSuspects()
generateActualTimeline()
generateClueMatrix()
generateRedHerrings()
generateChapterPlan()
generateChapterDraft()
checkMysteryFairness()
checkTimelineConsistency()
extractCaseStateChanges()
```

## 17. 结论

如果只针对现代推理小说，Agent 的核心应从“长篇写作”转为：

- 案件真相设计。
- 时间线管理。
- 线索公平性管理。
- 误导设计。
- 侦探推理路径。
- 逻辑漏洞检查。

推荐路线：

```text
先设计真相 -> 再倒推线索 -> 再安排误导 -> 再规划章节 -> 最后生成正文
```

一句话总结：

```text
现代推理小说 Agent 的核心不是 Writer Agent，而是 Case Architect + Logic Checker。
```

真正要做的产品不是“AI 帮我写一章”，而是：

```text
AI 帮我设计一个经得起读者复盘的案件。
```
