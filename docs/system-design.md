# 推理小说 Agent 系统设计（简化版）

这个 Agent 面向中长篇推理小说创作。用户输入关键词，例如“现代、刑警、连环杀人、宗教仪式、曲折、反转”，系统通过几轮确认，先设计一个经得起复盘的案件，再生成章节和正文。

核心目标只有一个：

```text
先把案件做严谨，再把故事写好看。
```

## 1. 产品范围

第一版只做：

- 单本中长篇。
- 一个核心案件。
- 多轮确认。
- 结构化案件档案。
- 章节大纲。
- 正文生成。
- 逻辑复盘检查。

暂不做：

- 多案件刑侦单元剧。
- 500 章长连载。
- 世界观资料库。
- 多主线群像管理。
- 复杂工作流编排。
- 向量库、Redis、任务队列。

这些可以以后加。第一版先证明：系统能不能帮用户设计出一个公平、严谨、可复盘的推理案件。

## 2. 基本流程

```text
关键词输入
  -> 方向确认
  -> 案件真相确认
  -> 嫌疑人与线索确认
  -> 章节大纲确认
  -> 正文生成
  -> 逻辑复盘与修订
```

每一步都要让用户确认或选择，不要一次性生成完整小说。

## 3. 三个角色

不用拆成很多 Agent。第一版只需要三个角色，代码上可以只是三个 prompt 或三类函数。

### 3.1 主持人

负责和用户交互，把模糊关键词变成明确方向。

它要问清楚：

- 故事背景：现代都市、县城、校园、封闭空间等。
- 主角身份：刑警、侦探、记者、法医、普通人等。
- 案件类型：谋杀、失踪、旧案重查、连环杀人等。
- 风格倾向：社会派、本格、悬疑、黑暗、冷峻等。
- 尺寸：约多少章、每章大概多长。
- 底线：哪些内容不写。

### 3.2 案件设计器

负责设计案件底层逻辑，不直接写正文。

它产出：

- 真相。
- 凶手。
- 动机。
- 作案方式。
- 掩盖方式。
- 嫌疑人。
- 时间线。
- 线索。
- 误导。
- 最终揭示。

### 3.3 审稿器

负责找漏洞。

它只检查这些问题：

- 凶手有没有作案时间。
- 作案方式是否现实可执行。
- 关键线索是否提前出现。
- 侦探是否提前知道未知信息。
- 证据是否足以支持结论。
- 嫌疑人说谎是否有理由。
- 误导是否最终解释清楚。
- 时间线是否冲突。
- 结尾是否能回扣前文。

## 4. 案件档案

第一版只维护一个 `NovelState`，不用拆很多表。

```json
{
  "brief": {
    "keywords": ["现代", "刑警", "连环杀人", "宗教仪式", "反转"],
    "length": "30-60章",
    "style": "冷峻、社会派、本格结合",
    "limits": ["不写超自然真相"]
  },
  "case": {
    "surface_mystery": "读者最初看到的谜面",
    "truth": "真实发生的案件过程",
    "culprit": "真凶",
    "motive": "真实动机",
    "method": "作案方式",
    "cover_up": "掩盖方式",
    "final_twist": "最终反转"
  },
  "characters": [],
  "timeline": [],
  "clues": [],
  "chapters": [],
  "confirmations": [],
  "quality_reports": []
}
```

这个对象是逻辑模型，不要求一次全部放进上下文。生成单章时只取：

- `brief`
- `case`
- 当前章节计划
- 相关人物、时间线和线索
- 已写章节摘要

完整正文和历史版本放在持久化层里，按需读取。

## 5. 最小字段

### 5.1 人物

```json
{
  "name": "林澈",
  "role": "刑警 / 嫌疑人 / 被害人 / 证人",
  "relationship": "与核心案件的关系",
  "motive": "表面动机或真实动机",
  "secret": "隐藏信息",
  "lie": "说谎点",
  "truth_status": "culprit / innocent / accomplice / victim / witness"
}
```

### 5.2 时间线

```json
{
  "time": "21:35",
  "location": "地下停车场",
  "actual_event": "真实发生的事",
  "claimed_event": "角色声称发生的事",
  "people": ["真凶", "被害人"],
  "reader_knows_at": "第几章",
  "detective_knows_at": "第几章"
}
```

### 5.3 线索

```json
{
  "id": "c001",
  "description": "未发送短信",
  "first_seen": "第3章",
  "surface_meaning": "看似指向前妻",
  "real_meaning": "实际指向备用手机",
  "payoff": "第26章",
  "fair": true
}
```

### 5.4 章节

```json
{
  "chapter": 12,
  "purpose": "强化错误嫌疑人，同时埋下真相线索",
  "new_clues": ["c001"],
  "misdirection": "让读者怀疑前妻",
  "must_not_reveal": "备用手机属于真凶"
}
```

## 6. 用户确认点

每次只确认一个层级，避免用户被一大坨设定砸中。

确认不是简单问“是否满意”。每轮都用同一个协议：

```text
给出 2 到 3 个可选方案
  -> 用户选择、改写或要求重来
  -> 系统总结本轮决定
  -> 用户确认
  -> 写入 NovelState.confirmations
  -> 生成下一个层级
```

确认记录格式：

```json
{
  "step": "case_direction",
  "status": "confirmed",
  "summary": "现代都市刑侦，连环仪式杀人，社会派底色，本格解谜结构",
  "locked_fields": ["brief.keywords", "brief.style", "case.surface_mystery"],
  "created_at": "2026-07-07T12:00:00Z"
}
```

`locked_fields` 里的内容后续不能被静默改掉。需要修改时，必须生成新版本并在界面提示“这会影响后续线索和章节计划”。

### 6.1 确认界面

每轮页面只放三类操作：

| 操作 | 含义 |
| --- | --- |
| 选择方案 | 接受某个方向，进入确认摘要 |
| 修改要求 | 用户补充偏好，例如“更黑暗一点，不要超自然” |
| 重新生成 | 当前方向不对，丢弃本轮候选 |

确认摘要要短，只列真正会影响后续生成的决定。

### 6.2 确认 API

第一版只需要一个通用接口：

```text
POST /novels/:id/confirm
```

请求：

```json
{
  "step": "case_direction",
  "decision": "选择 A，但把宗教元素改成伪装",
  "locked_fields": ["brief.style", "case.surface_mystery"]
}
```

服务端做三件事：

1. 读取当前版本。
2. 把确认写入 `confirmations`。
3. 生成新版本并更新 D1 的 `current_version`。

不要为每一轮单独设计一个确认接口，先用 `step` 区分。

### 第 1 轮：方向

系统给 2 到 3 个故事方向，让用户选。

示例：

```text
A. 冷峻刑侦：连环仪式杀人背后是旧案复仇。
B. 本格封闭：宗教团体内部出现不可能犯罪。
C. 社会派悬疑：仪式只是伪装，真相指向权力遮掩。
```

### 第 2 轮：核心案件

确认：

- 谁死了。
- 表层谜面是什么。
- 真凶是谁。
- 真实动机是什么。
- 最终反转是什么。

真凶可以先不告诉用户，但系统内部必须固定。否则后面线索会散。

### 第 3 轮：嫌疑人与误导

确认：

- 3 到 6 个主要嫌疑人。
- 每个人的动机。
- 每个人的秘密。
- 每个人为什么看起来可疑。
- 最后如何解释清楚。

### 第 4 轮：线索与时间线

确认：

- 真实时间线。
- 表层时间线。
- 关键线索。
- 每条线索首次出现和最终回收。

### 第 5 轮：章节大纲

确认：

- 章节数量。
- 每个阶段的功能。
- 哪些章节埋线索。
- 哪些章节制造误导。
- 哪些章节反转。
- 哪章揭示真相。

## 7. 章节结构

默认用 30 到 60 章。不要第一版就支持任意超长结构。

30 章示例：

| 章节 | 功能 |
| --- | --- |
| 1-3 | 案发、谜面、主角进入案件 |
| 4-8 | 嫌疑人登场，第一批线索出现 |
| 9-12 | 强化第一个错误嫌疑人 |
| 13-16 | 第一次推理失败，旧线索被重新解释 |
| 17-21 | 隐藏关系和真实动机浮出 |
| 22-25 | 时间线矛盾集中爆发 |
| 26-28 | 回收线索，逼近真相 |
| 29 | 最终推理与反转 |
| 30 | 余波与主题落点 |

## 8. 生成正文的规则

正文生成必须受案件档案约束。

硬规则：

1. 不新增未经确认的关键证据。
2. 不让侦探知道尚未发现的信息。
3. 不泄露 `must_not_reveal`。
4. 不用巧合破案。
5. 不让角色无理由说谎。
6. 本章必须完成 `purpose`。
7. 新增事实必须写回 `NovelState`。

正文不是自由续写，而是把章节计划写成小说。

## 9. 复盘检查

每次生成案件、章节大纲或正文后，都跑一次最小检查。

```json
{
  "pass": false,
  "problems": [
    {
      "severity": "high",
      "issue": "第29章推理依赖备用手机，但备用手机第一次出现也是第29章。",
      "fix": "在第7章现场勘查时加入备用手机的异常信号记录。"
    }
  ]
}
```

检查清单：

- 时间线能否对上。
- 凶手行动路径是否可行。
- 关键线索是否提前出现。
- 每条关键线索是否有回收。
- 红鲱鱼是否解释清楚。
- 侦探推理是否跳步。
- 证据链是否过弱。
- 反转是否推翻前文事实。

## 10. 技术方案

沿用当前仓库的轻量栈即可：

- Vue：写作界面和案件档案编辑。
- Hono/Cloudflare Worker：生成 API。
- Cloudflare D1：保存小说项目、当前版本、章节索引和生成记录。
- Cloudflare R2：保存 `NovelState` 快照、章节正文和导出文件。
- LLM Structured Output：约束案件档案、章节和检查报告。

暂不引入：

- PostgreSQL。
- pgvector。
- Redis/BullMQ。
- 多 Agent 调度框架。
- 独立状态机系统。

### 10.1 持久化方式

Cloudflare Worker 本身是无状态的，不能把小说进度放在内存里。每次用户确认或生成一章，都写一个新版本。

最小存储模型：

```text
D1
  novels(id, title, current_version, created_at, updated_at)
  versions(id, novel_id, kind, r2_key, created_at)
  chapters(id, novel_id, chapter_no, title, status, r2_key, updated_at)

R2
  novels/{novel_id}/state/{version}.json
  novels/{novel_id}/chapters/{chapter_no}/{version}.md
  novels/{novel_id}/exports/{version}.md
```

D1 只放索引和小字段；R2 放大 JSON、正文和导出文件。这样查询列表快，正文也不会塞进关系表。

### 10.2 写入流程

```text
读取 current_version
  -> 从 R2 取 NovelState
  -> 生成下一步内容
  -> 审稿器检查
  -> 写入新的 R2 快照
  -> 更新 D1 current_version 和章节索引
```

每次写入都产生新版本，不覆盖旧版本。用户可以回滚到任意一次确认后的状态。

### 10.3 什么时候加 Durable Object

第一版可以先不用 Durable Object。如果同一本小说可能同时触发多个生成任务，再加：

```text
一个 novel_id -> 一个 NovelSession Durable Object
```

它只负责串行化同一本小说的写入，避免两个请求同时基于旧版本生成并互相覆盖。正文和状态仍然落到 D1/R2，不只放在 Durable Object 内存里。

KV 不适合作为主存储。它可以做只读缓存，但小说状态、章节正文和版本记录应以 D1/R2 为准。

## 11. MVP API

第一版只需要这些能力：

```text
createBrief(keywords)
proposeDirections(brief)
confirmStep(novelId, step, decision)
createCase(direction)
reviseCase(case, userFeedback)
createChapterPlan(novelState)
draftChapter(novelState, chapterNumber)
checkNovelState(novelState)
loadNovel(novelId)
saveNovelVersion(novelId, novelState)
```

这些函数可以覆盖完整闭环。

## 12. 最小页面

第一版页面不要做复杂看板。

只需要三块：

| 区域 | 内容 |
| --- | --- |
| 左侧 | 用户输入、确认问题、版本记录 |
| 中间 | 当前生成结果：方向、案件、章节或正文 |
| 右侧 | 案件档案摘要和复盘问题 |

可视化矩阵、嫌疑值曲线、信息释放曲线都先不做。等用户真的需要定位复杂长篇问题时再加。

## 13. 结论

第一版的核心不是“很多 Agent”，而是一个稳定闭环：

```text
用户确认 -> 更新案件档案 -> 生成下一步 -> 复盘检查 -> 修订
```

只要这个闭环可靠，系统就能服务推理小说创作。复杂数据库、多 Agent、任务队列和高级看板，都可以等第一个严谨案件跑通后再加。
